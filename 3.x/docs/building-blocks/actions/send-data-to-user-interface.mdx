**Why is it useful?** This action enables real-time updates and seamless communication between your backend process and frontend application, making it ideal for live data streaming, instant notifications, and dynamic UI updates.

## Configuring a Send data to user interface action

You can configure this action through the **FlowX Designer** using multiple options. To set up a Send data to user interface action, navigate to the **Actions** tab at the [task node level](../../flowx-designer/managing-a-process-flow/adding-a-new-node). The following configuration sections are available:

* [Action Edit Configuration](#action-edit)
* [Back in steps (for Manual actions)](#back-in-steps)
* [Parameters Configuration](#parameters)
* [Data to send (for Manual actions)](#data-to-send)

### Action Edit

* **Name** - A unique identifier used internally to distinguish between different node actions in the process. We recommend implementing a standardized action naming convention for easier process action discovery
* **Order** - Determines the execution sequence when multiple actions are configured on the same node
* **Timer expression** - Enables action delay configuration using the [**ISO 8601 duration format**](https://www.w3.org/TR/NOTE-datetime) (example: `PT30S` for a 30-second delay)
* **Action type** - Must be set to Send data to user interface
* **Trigger type** (Automatic/Manual) - Determines if the action triggers automatically (when process flow reaches the step) or manually (user-triggered); typically set to automatic
* **Required type** (Mandatory/Optional) - Automatic actions are always mandatory; manual actions can be either mandatory or optional
* **Repeatable** - Enable this option to allow multiple action triggers
* **Autorun Children** - When enabled, child actions (marked as mandatory and automatic) execute immediately after parent action completion

### Back in steps

* **Allow BACK on this action** - Enables the back-in-process functionality, allowing users to return to previous steps and redo actions. For detailed information, see [Moving a token backwards in a process](../../flowx-designer/managing-a-process-flow/moving-a-token-backwards-in-a-process).

### Parameters

Essential configuration fields for this action type include:

* **Message Type** - Set to **Default** for basic data transmission (defaults to **data** message type)

<Warning>
To initiate a new process using **Send data to user interface**, set the **Message Type** to **Action** and define a **Message** using this format:
</Warning>

```json
{
  "processName": "demoProcess",
  "type": "START_PROCESS_INHERIT",
  "clientDataKeys":["webAppKeys"],
  "params": {
     "startCondition": "${startCondition}",
     "paramsToCopy": []
     }
}
```

<Info>
* `paramsToCopy` - Select which parent process parameter keys should be copied to the subprocess
* `withoutParams` - Specify which parent process parameter keys should be excluded when copying values to the subprocess
</Info>

* **Message** - Define the data payload as a JSON object, supporting both constant values and process instance data values
* **Target Process** - Specify the destination process instance for the message - choose between **Active process** or **Parent process**

<Info>
When configuring this action in a [**subprocess**](../node/subprocess-run-node), you can direct messages to the parent process by selecting **Target Process: Parent process**.
</Info>

### Data to send

* **Keys** - Used for data validation when receiving frontend action data (for more details, see the [User task configuration](../node/user-task-node) section)

<Warning>
The **Data to send** configuration is only available when the action **trigger type** is set to **Manual**.
</Warning>

### Send update data example

Here's how to send the latest value from the `application.client.firstName` key in your [process instance](../process/process-instance) data to the frontend application:

1. Create a new **Send data to user interface** action
2. Configure **Message Type** as **Default** (this uses the default `data` type)
3. Define the **Message** with your data payload:
   * `{ "name": "${application.client.firstName}" }`
4. Select your desired **Target Process**
