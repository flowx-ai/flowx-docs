## Overview

<Frame>
![Text area component example](https://s3.eu-west-1.amazonaws.com/docx.flowx.ai/post41/text_area41.png)
</Frame>

Text areas are ideal for collecting longer user input in forms and conversational interfaces. They provide a larger input space and can automatically expand to fit content.

<Info>
Unlike single-line input fields, text areas are designed specifically for multi-paragraph content and preserve line breaks entered by users.
</Info>

## Configuration Options

### Core Settings

The text area component can be configured through several key settings:

<AccordionGroup>
<Accordion title="Process Data Key">
The process data key creates a binding between the text area and your process data. This allows the entered text to be:

- Used in [decision nodes](../../../node/exclusive-gateway-node)
- Processed by [business rules](../../../actions/business-rule-action/business-rule-action)
- Sent to [integrations](../../../node/message-send-received-task-node#from-integration)

<Warning>
Choose a descriptive and unique key name to avoid conflicts with other form elements.
</Warning>
</Accordion>

<Accordion title="Basic Properties">
Configure the following basic properties:

- **Label**: Text label shown above the text area
- **Placeholder**: Hint text shown when the field is empty
- **Has Clear**: Adds a clear button to empty the field
- **Helper Text**: Additional context shown below the field (can be hidden in an info point)
- **Update on Blur**: Whether to update the value when focus leaves the field

<Info>
Helper text is useful for providing input guidelines or requirements to users.
</Info>
</Accordion>

<Accordion title="Default Value">
Set an initial value for the text area through the datasource configuration:

1. Open the Datasource tab
2. Enter the default text value
3. Optionally enable "Computed Value" to use a JavaScript expression

<Tip>
Use computed values when the default text needs to be dynamically generated.
</Tip>
</Accordion>
</AccordionGroup>

### Dynamic Behavior

#### Hide/Show Logic

Control the text area's visibility using JavaScript expressions:

```javascript
// Example: Hide text area if switch is off
${application.client.hasHouse} === false
```

<Frame>
![Hide expression example](https://s3.eu-west-1.amazonaws.com/docx.flowx.ai/post41/hide_text_area1.png)
</Frame>

<Info>
The hide expression is evaluated whenever referenced values change in your process data.
</Info>

#### Enable/Disable Logic

Disable the text area conditionally:

```javascript
// Example: Disable when contact method is SMS
${application.client.contact} == "prfS"
```

<Warning>
Always configure the same expression under both the disabled condition and hide expression for consistent behavior.
</Warning>

### Event Handling

Configure UI actions to respond to text area events:

- **Event Type**: Currently supports `CHANGE`
- **Action Types**: Various actions like updating process data or triggering validators

<Tip>
See [UI Actions](../../ui-actions) for detailed configuration examples.
</Tip>

## Styling Configuration

### Sizing Options

<Tabs>
<Tab title="Web">
#### Width Control
- **fill**: Expands to container width
- **fixed**: Uses specified pixel width
- **auto**: Sizes to content width

#### Height Control
- **fixed**: Set exact height in pixels
- **auto**: Grows with content

#### Row Configuration
- **Min Rows**: Minimum visible rows
- **Max Rows**: Maximum rows before scrolling
</Tab>

<Tab title="iOS">
- Uses points (pt) for fixed height
- Inherits most web styling options
</Tab>

<Tab title="Android">
- Uses density-independent pixels (dp)
- Inherits most web styling options
</Tab>
</Tabs>
